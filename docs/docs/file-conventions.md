# File Conventions

This page lists the filenames and directories that evjs treats as framework
conventions. Files outside these convention roots are ordinary application
modules unless they are imported by a convention file.

## Convention Roots

| File or directory | Convention area | Meaning |
| --- | --- | --- |
| `src/pages/**/*.{ts,tsx,js,jsx}` | Client pages | Client page route discovery for SPA and MPA output. |
| `src/pages/**/*.html` beside a same-basename page route | MPA pages | MPA page-specific HTML template, such as `about.html` for `about.tsx` or `users/index.html` for `users/index.tsx`. |
| `<routing-dir-parent>/layout/index.tsx` | SPA layouts | Optional external SPA root layout auto-discovered by convention. |
| `src/pages/<segment>/**/layout.{ts,tsx,js,jsx}` | SPA layouts | Nested SPA route layout inside the page route tree. |
| `src/pages/**/error.{ts,tsx,js,jsx}` | SPA boundaries | SPA error boundary for the nearest route directory scope. |
| `src/pages/**/not-found.{ts,tsx,js,jsx}` | SPA boundaries | SPA not-found boundary for the nearest route directory scope. |
| `<routing-dir-parent>/route-types.d.ts` | Generated output | SPA navigation type declaration generated by evjs. |
| `*.server.{ts,tsx,js,jsx}` files with `"use server";` | Server functions | Recommended server-function naming convention. |
| `src/apis/**/*.{ts,tsx,js,jsx}` | Server file routes | Server file routes discovered by default. |
| `src/middleware.{ts,tsx,js,jsx}` | Server middleware | Global server middleware. |
| `src/apis/**/middleware.{ts,tsx,js,jsx}` | API route middleware | API route middleware for server file routes. |

The default client route directory is `./src/pages`. The default server file
route directory is `./src/apis`. Configure a different client route directory
with `routing.dir`, configure a different server file route directory with
`server.routing.dir`.

## Segment Rules

Page routes, server file routes, and API route middleware use the same
path segment rules:

| Pattern | Result |
| --- | --- |
| `index.*` | Directory root route. |
| `$param.*` | Dynamic segment. Server file routes convert it to Hono `:param`. |
| `(group)` | Pathless route group for organization. It does not add a URL segment. |
| `_private.*` or `_private/` | Ignored private module or directory. |
| `.hidden.*` or `.hidden/` | Ignored hidden module or directory. |

Static route segments must use lowercase URL-safe characters: lowercase
letters, numbers, `.`, `_`, `-`, or `~`. Dynamic param names must be JavaScript
identifiers after `$`, such as `$userId` or `$team_id`.

The following are rejected:

- bracket routes such as `[id].tsx`;
- catch-all routes such as `$...slug.tsx`;
- optional params such as `$slug?.tsx`;
- empty dynamic params such as `$.tsx`;
- reserved dynamic params such as `$__proto__.tsx`, `$constructor.tsx`,
  `$prototype.tsx`, or `$_splat.tsx`;
- duplicate dynamic params within one path, such as
  `teams/$teamId/users/$teamId.tsx`;
- duplicate paths, such as `users.tsx` beside `users/index.tsx`;
- duplicate dynamic shapes, such as `users/$id.tsx` beside
  `users/$userId.tsx`.

Routes must follow the file shape. evjs does not provide alternate filename
dialects for catch-all, optional, or bracket routes.

## Ignored Support Files

Under route directories, evjs ignores support files so they can be colocated
with routes:

| Pattern | Meaning |
| --- | --- |
| `*.d.ts` | Type declarations. |
| `*.test.*` and `*.spec.*` | Tests. |
| `*.story.*` and `*.stories.*` | Storybook stories. |
| `*.client.*` | Client-only modules. |
| `*.server.*` | Server-only modules. |
| files without `.ts`, `.tsx`, `.js`, or `.jsx` | Non-source assets or metadata, except same-basename MPA `.html` templates. |

Use `_`-prefixed files or folders for route-local helpers that still use source
extensions. For example, `src/pages/users/_format.ts` and
`src/pages/users/_components/Card.tsx` are not URL routes.

## Client Page Routes

Client page routes live under `routing.dir`:

```text
src/pages/index.tsx              -> /
src/pages/about.tsx              -> /about
src/pages/users/index.tsx        -> /users
src/pages/users/$userId.tsx      -> /users/$userId
src/pages/(marketing)/about.tsx  -> /about
```

Each discovered page file must default-export a React component. Page modules
may also export literal rendering metadata such as `render`, `hydrate`,
`prerender`, and `rsc`.

SPA route layouts use `layout.*` files below a route segment:

```text
src/pages/dashboard/layout.tsx   -> wraps /dashboard descendants
```

The external SPA root layout has one auto-discovered convention:

```text
src/layout/index.tsx
```

`src/pages/layout.tsx`, root layout aliases such as `src/layout.tsx`, and
route-layout directory aliases such as `src/pages/dashboard/layout/index.tsx`
are rejected by convention. Configure `routing.conventions.layout` explicitly
only when the application shell intentionally lives elsewhere. MPA mode does
not consume framework layouts.

SPA route boundaries use dedicated convention files:

```text
src/pages/error.tsx             -> root error boundary
src/pages/not-found.tsx         -> root not-found boundary
src/pages/dashboard/error.tsx   -> /dashboard scoped error boundary
```

`error.*` and `not-found.*` are scoped by directory and inherited by descendant
page and layout routes until a closer boundary exists. Boundary modules must
default-export a React component. These SPA router conventions are not consumed
in MPA mode, where the filenames remain ordinary page routes.

In MPA mode, a page route can use a colocated HTML template with the same
basename as the page module:

```text
src/pages/about.tsx        -> /about, template src/pages/about.html
src/pages/about.html
src/pages/users/index.tsx  -> /users, template src/pages/users/index.html
src/pages/users/index.html
```

Routes without a colocated template use the global `index.html` template by
default. If every MPA route has a colocated template, `index.html` is not
required for those routes.

SPA mode writes generated route types to `<routing-dir-parent>/route-types.d.ts`.
Do not edit that file, import it from app code, or copy it into templates.

## Server Functions

Server functions are detected by the `"use server";` directive, not by a
directory. Use `.server.ts` or `.server.tsx` filenames so route discovery
ignores colocated server-only files:

```ts
// src/apis/users.server.ts
"use server";

export async function listUsers() {
  return [];
}
```

Rules:

- the directive must be at the top of the module;
- the module must export at least one named callable function;
- default exports, runtime re-exports, generators, async generators, and
  exported non-function runtime values are not supported;
- type-only exports are ignored by the runtime transform;
- reachable server functions can be called from browser code through the built-in
  transport; app code does not need to write endpoint or proxy files.

## Server File Routes

Server file routes are enabled by default. The URL comes from the file path
under `server.routing.dir`; there is no `prefix` option.

```text
src/apis/index.ts              -> /
src/apis/health.ts             -> /health
src/apis/users.ts              -> /users
src/apis/users/index.ts        -> /users
src/apis/users/$userId.ts      -> /users/:userId
src/apis/(internal)/health.ts  -> /health
src/apis/api/users.ts          -> /api/users
```

A file becomes a route only when it exports at least one uppercase HTTP method:
`GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, or `OPTIONS`.

```ts
// src/apis/api/posts.ts
export const GET = async () => Response.json([]);

export const POST = async (request) => {
  const body = await request.json();
  return Response.json(body, { status: 201 });
};
```

Server route candidates may export only uppercase HTTP methods. Move shared
helpers to ignored private files such as `_schema.ts` or to modules outside the
route tree.

The following are rejected in server route candidates:

- default exports;
- `middleware` or `middlewares` exports;
- helper exports or unsupported runtime exports;
- lowercase HTTP method exports such as `get`;
- method suffix files such as `posts.get.ts`;
- `route.ts` sentinel files;
- duplicate paths or duplicate dynamic shapes.

Files without HTTP method exports are ignored, so colocated helpers such as
`schema.ts`, `db.ts`, and `types.ts` can live in the route tree.

## Server Middleware

evjs has two server middleware scopes. Middleware files default-export one
Hono-compatible middleware function. They must not export named values or
matcher configuration.

Global server middleware wraps server runtime requests, including
server file routes, server functions, SSR, PPR, and RSC:

```ts
// src/middleware.ts
import type { MiddlewareHandler } from "@evjs/ev/server-context";

const middleware: MiddlewareHandler = async (ctx, next) => {
  await next();
  ctx.header("x-server", "evjs");
};

export default middleware;
```

Global server middleware:

```text
src/middleware.ts
```

API route middleware:

```text
src/apis/middleware.ts
src/apis/api/middleware.ts
src/apis/api/admin/middleware.ts
src/apis/(admin)/middleware.ts
```

Execution order is:

1. global server middleware;
2. API route middleware from parent directory to child directory;
3. the HTTP method handler.

API route middleware applies only to descendant server file routes; for example,
`src/apis/api/middleware.ts` covers `src/apis/api/users.ts`
but not the flat sibling `src/apis/api.ts`.
